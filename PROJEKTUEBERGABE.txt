TASTATUR-QUIZ (Tippen-Modus)
-----------------------------
- 3 Fehlversuche erlaubt bevor Score-Abzug + automatische Lösungsanzeige
- 1. + 2. Fehlversuch: Box leuchtet 1,5s rot, "Nicht richtig" (zentriert) erscheint
  und verschwindet mit der roten Box → kein Score-Abzug
- 3. Fehlversuch: Score -1, Lösung wird automatisch gezeigt (phase: "aufgedeckt")
- Lösungstext: quiz-frage-text Klasse (1.5rem, fett), minHeight:"44px" für stabile Box
- "Multiple Choice"-Button (wechsleZuMC) ist weggefallen
- State: quiz.aktuelleFehlversuche, quiz.falschFlash

DIKTAT-QUIZ
-----------
- Lösung zeigt nur: **fettes Wort** – Übersetzung (kein "Deine Eingabe"-Label, kein Hint)
- Fragebox: Übersetzungshinweis immer im DOM (visibility:hidden/visible) → keine Größensprünge
- Stimmenfilter: Novelty/Effekt-Stimmen (Boing, Jester, Zarvox, Fred, etc.) ausgefiltert
- Button "Andere Stimme" (früher "Nochmal") wechselt die Stimme alle 2 Plays

STATISTIK-TAB
-------------
- Sticky Listen-Header (top:headerH): Listenauswahl, Zurücksetzen, Alle, Ausklappen
- Graph: Quiz-Score (grün, durchgezogen) + Diktat-Score (orange, gestrichelt)
- Graph-Toggle: "Inkl. unbeantwortete" / "Nur beantwortete" (State: statistikGraphOhneUnbeantwortet)
  → Bei "Nur beantwortete": jede Linie plottet nur Vokabeln mit vorhandenem Fortschritt
- Kennzahlen: Gesamt, Nie abgefragt, Score positiv, Score negativ, Ø Score
- Filter: Alle, Nie, Neg, Null, Pos
- Sortierung: Score ↑, Score ↓, Streak, Datum, A→Z


================================================================
SCORING-LOGIK
================================================================
Tastatur/MC/Karte:
  Richtig:    +1, Streak +1 (ab Streak 3: Score min. 2)
  Falsch:     Streak 0, Score -1/-2/-3 je sessionFalschAnzahl
              NEU: erst beim 3. Fehlversuch wird Score abgezogen + Lösung gezeigt
  Aufdecken:  Streak 0, aufgedecktStreak +1 (schwer: -1; nach 3x: Score=-100)

Diktat:
  Richtig:    +1
  Aufdecken:  -1

State-Felder im quiz-Objekt:
  sessionFalsch: {vokId: count}   — Fehlanzahl je Vokabel in dieser Session
  aktuelleFehlversuche: 0-2       — Fehlversuche für die aktuelle Frage
  falschFlash: bool               — true während des 1,5s-Rotblitzes

GEPLANTE ÄNDERUNGEN FÜR SESSION 9

KONZEPT: STICKY-ON-SCROLL ELEMENTE
------------------------------------
Elemente die im normalen Content-Flow liegen, aber beim Scrollen am
Header anstossen und dort kleben bleiben (standard CSS position:sticky
mit top:headerH, AUSSERHALB des headerRef-Wrappers, direkt in .app).
Beim Zurückscrollen lösen sie sich wieder ab.

ÄNDERUNG 1 – "Quiz starten"-Button: sticky-on-scroll im Content
----------------------------------------------------------------
Aktuell: Button ist im headerRef-Wrapper (immer sichtbar, aber kein Scroll-Verhalten)
Neu: Button aus headerRef entfernen, stattdessen im Content-Flow platzieren:
  - Position: nach der ausgeklappten Listenliste, vor den Slots
  - style: position:"sticky", top:headerH, zIndex:8, background:"#fff"
  - Padding + borderBottom damit er beim Kleben sauber aussieht
  - Wenn Listen eingeklappt: Button ist erstes Content-Element → klebt sofort
  - Wenn Listen ausgeklappt: Button scrollt mit hoch bis er am Header anstösst
  - Button zeigt verfuegbar-Anzahl, disabled wenn !kannStarten || verfuegbar===0
  - kannStarten + verfuegbar werden bereits im Quiz-IIFE berechnet → direkt verfügbar
  - Gleichzeitig: IIFE im headerRef für den Button ENTFERNEN

ÄNDERUNG 2 – Statistik: Ø Score sticky-on-scroll
-------------------------------------------------
Die Ø-Score-Zeile (derzeit normaler Content) soll sticky werden:
  style: position:"sticky", top:headerH, zIndex:8, background:"#fff"
Damit klebt der Durchschnittsscore beim Scrollen am Header.
Hinweis: headerH im Statistik-Tab = Höhe von Topbar + Tabs + statistik-listen-header

ÄNDERUNG 3 – Statistik: "Nach oben"-Button sticky am Ø-Score
-------------------------------------------------------------
Einen "↑ Nach oben"-Button zum Ø-Score-Element hinzufügen (rechts bündig).
Klick: scrollt den .app-Container per scrollTo({top:0, behavior:"smooth"}) nach oben.
Der Button klebt mit dem Ø-Score am Header → ist beim Scrollen immer sichtbar.
Scroll-Container ermitteln: document.querySelector('.app') oder appRef falls vorhanden.

ÄNDERUNG 4 – Statistik: Sortierung mit Richtungs-Toggle
--------------------------------------------------------
Aktuell: 5 Buttons (Score↑, Score↓, Streak, Datum, A→Z) — Score doppelt
Neu: 4 Buttons (Score, Streak, Datum, A→Z) — jeder Button togglet Richtung
  State-Änderung:
    statistikSort (string "score-asc") → statistikSortKey + statistikSortDir
    ODER: statistikSort bleibt als kombinierter String: "score-asc" / "score-desc" etc.
    Empfehlung: statistikSort als kombinierter String beibehalten, nur UI ändern
  UI:
    Aktiver Button zeigt Pfeil: "Score ↑" oder "Score ↓"
    Klick auf aktiven Button: Richtung umkehren
    Klick auf inaktiven Button: aktivieren (Default-Richtung: asc für Score, desc für Streak/Datum)
  Alle Buttons: {key:"score", label:"Score"}, {key:"streak",...}, {key:"datum",...}, {key:"alpha",...}
  Default-Richtungen beim ersten Klick:
    score → asc (schlechteste zuerst), streak → desc (beste zuerst),
    datum → desc (neueste zuerst), alpha → asc (A→Z)

ÄNDERUNG 5 – Quiz-Listen-Toggle ohne Scroll-Sprung
---------------------------------------------------
Problem: Wenn der User weit unten im Quiz-Tab scrollt und dann
"Ausklappen/Einklappen" klickt, springt der Viewport weil sich die
Höhe des Listen-Bereichs ändert, der sich ÜBER dem Viewport befindet.

Lösung: Scroll-Offset-Korrektur im Click-Handler:
  1. Ref auf den Listen-Container (listenContainerRef)
  2. Vor dem Toggle: aktuelle Scroll-Position + getBoundingClientRect() messen
  3. Nach dem Toggle (useLayoutEffect oder requestAnimationFrame):
     prüfen ob der Listen-Bereich über dem Viewport liegt
  4. Wenn ja: Scroll-Offset um die Höhendifferenz korrigieren

Alternativer Ansatz: CSS overflow-anchor
  .app { overflow-anchor: auto; } (Browser-Standard-Scroll-Anchoring aktivieren)
  Wenn .app der Scroll-Container ist: möglicherweise reicht das bereits.
  Testen ob Browser-Standard-Anchoring greift; falls nicht → JS-Lösung.

Implementierungsreihenfolge:
  Zuerst CSS-Ansatz testen, falls unzureichend: JS mit useLayoutEffect
